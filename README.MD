## Explanation

### Database Structure

I first looked at the data `cell-count.csv` and thought about what tables would make sense to be modeled in this scenario.

I see that every row has a project ID, subject ID, info about the subject, and a sample alongside the sample results.

Quickly scanning through the data I saw that each project can have many subjects, each of which can have many samples.

Hence, a table for projects, subjects, and samples with one-to-many relationships from left to right seems to be most fitting.

This design avoids duplicating subject metadata (age, sex, condition, treatment, response) on every sample row, while still letting me join quickly from projects -> subjects -> samples for analysis.

This also scales well since if there are hundreds of projects and thousands of samples, queries just join on primary/foreign keys (`project_id`, `subject_id`) and filter on columns like `condition`, `treatment`, `sample_type`, all of which are easy to index in SQLite.

If the dataset grew more complex as well (new columns), we can also just add a new table and key it by `sample_id`, further chaining the joins / foreign keys.

If the sheer volume of data becomes astronomical, we can consider sharding by `project_id`, for example.

### Code Structure

- `cell-count.csv`: Raw input data.

- `load_data.py`: Script that loads all rows from `cell-count.csv` and builds the SQLite database in accordance to the schema described above (`projects`, `subjects`, `samples`).

- `cell_counts.db`: SQLite database generated by `load_data.py`.

- `dashboard.py`: Single Flask app that serves all analysis. Includes the following routes:
  - `/part2`: Summary table of per-sample cell population frequencies.
  - `/part3`: Responders vs non-responders (melanoma + miraclib, PBMC) with boxplots and Mannâ€“Whitney p-values.
  - `/part4`: Baseline subset (melanoma, PBMC, time=0, miraclib) with aggregate counts and an interactive, filterable table.
- `templates/`: Simple HTML templates for each dashboard page (`index.html`, `part2.html`, `part3.html`, `part4.html`).
- `Makefile`: `make setup` (install deps), `make pipeline` (load DB), `make dashboard` (run Flask server).
- `requirements.txt`: Python dependencies (Flask + plotting/stats stack).

I chose Flask for its minimal footprint. For the sake of this technical, I kept each part on its own page for easier grading since I want to avoid overengineering. If this were to become production and scale for more features on the dashboard, I'd split routes into blueprints, add a proper frontend framework (e.g. React), and introduce caching for heavy queries.
